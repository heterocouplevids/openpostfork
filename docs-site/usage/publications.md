# Publications

Publications store the source idea before it becomes platform-specific posts. Use them for launch notes, article summaries, campaign briefs, reusable assets, or anything an assistant, CLI script, or human editor should reference later.

## Create a publication

1. Open **Publications** from the app menu.
2. Add a short title.
3. Paste the source content.
4. Optionally add a source URL, goal, audience, and source media.
5. Create the publication.

The source can stay as a draft while you collect context. Mark it ready when it is approved for drafting.

## Compose from a publication

Open a publication and choose **Compose**. The composer links the source publication, pre-fills the first post with the source content when the composer is empty, and keeps the `publication_id` on saved drafts, scheduled posts, and threads.

Use **Clear** in the composer when a post should no longer be connected to the source publication.

## When to use publications

- Use a publication when one idea may become several platform-native versions.
- Use a normal post when you already know the final copy and destination.
- Use source media on the publication when the same assets should be available to web, CLI, and MCP workflows.

## Other workflows

- CLI users can manage the same source records with [`openpost publication`](/cli/publications).
- MCP clients can create and list source publications before drafting or scheduling posts.
